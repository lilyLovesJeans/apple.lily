1️⃣ 基本概念

git rebase 的作用是 把一個分支的修改「搬到」另一個分支的最前面。
換句話說，它不是把分支「合併」在一起，而是 改寫提交歷史，讓它看起來像是在目標分支之後直接做的修改。

簡單比喻：

  • 假設你在 main 分支上有提交 A → B → C

2️⃣ Rebase vs Merge

特性 Merge Rebase
歷史 保留所有分支歷史，會出現 merge commit 重新寫歷史，保持線性，不會產生 merge commit
視覺上 分支歷史會分叉 分支歷史線性，看起來像直接接在 main 後面
衝突解決 merge 時解決一次衝突 每個 commit 都可能需要解決衝突

4️⃣ 使用場景

  • 保持提交歷史乾淨：讓歷史看起來線性，不出現太多 merge commit。
  • 把 feature 分支更新到最新 main：在 push 前用 rebase，避免 merge commit。

⚠️ 注意：不要對已經推送到遠端的分支做 rebase，會改變 commit hash，可能導致別人 pull 出問題。

1️⃣ 情境 1：小組開發新功能（建議用 rebase）

情境：

  • 你在 feature/login 分支開發登入功能。
  • 你的同事在 main 分支做了更新（例如修正了 API URL）。
  • 你想把最新的 main 更新拿到你的 feature/login 分支上，保持分支乾淨。

做法：

效果：

  • 你的 feature 分支會「搬到最新 main 後面」，歷史線性。
  • 不會出現 merge commit，git log 乾淨。

✅ 適用時機：你還沒把 feature 分支推到遠端，或團隊允許改寫分支歷史。

2️⃣ 情境 2：完成 feature，要合併回 main（建議用 merge）

情境：

  • 你開發的 feature 分支已完成，經過測試。
  • 你要把 feature/login 分支合併回 main，保留開發過程的歷史。

做法：

效果：

  • 會產生一個 merge commit，把 feature 分支完整合併回 main。
  • 歷史保留原本分叉過程，方便追蹤誰在何時做了什麼。

✅ 適用時機：feature 分支已經推送給團隊，或你想保留分支歷史。

3️⃣ 情境 3：多人長期開發同一分支（混合策略）

  • 每個人先 rebase 自己的 feature 分支到最新 main，保持線性、避免衝突。
  • 最後再用 merge 將完成的 feature 分支合併回 main，保留完整歷史。

這樣可以兼顧「線性歷史」與「完整紀錄」。

💡 簡單記憶法：

  • Rebase = 更新你的分支（搬到最新 main 之後，乾淨線性）
  • Merge = 合併完成的分支回主線（保留歷史，出現 merge commit）
