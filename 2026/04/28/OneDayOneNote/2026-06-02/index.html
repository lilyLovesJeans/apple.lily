2026-06-02 部署到雲端, 資料庫要怎麼連?

我:
部署到雲端, 資料庫要怎麼連?
GPT:
當您將 Web API 部署到雲端（例如使用 Azure App Service）後，您的應用程式需要能夠正確地連接到資料庫。在雲端環境中，您有幾個不同的選擇來管理和連接資料庫。以下是部署 Web API 後連接資料庫的步驟：

1. 雲端資料庫選擇

  • Azure SQL Database（如果使用 SQL Server）
  • Azure Database for MySQL（如果使用 MySQL）
  • Azure Cosmos DB（適用於非關係型資料）
  • 其他雲端托管的資料庫服務

2. 設置資料庫連接字符串

無論您的資料庫是本地的還是雲端的，Web API 通常使用連接字符串來連接資料庫。在部署到 Azure App Service 之後，您需要確保您的應用程式可以找到正確的連接字符串來連接到資料庫。

  1. 在 Azure App Service 中配置連接字符串：

    • 登入 Azure 入口網站。

    • 在 Azure App Service 中，找到您的應用程式。

    • 點擊左側菜單中的 「設定」，選擇 「應用設定」（Application settings）。

    • 在「連接字符串」部分，您可以添加新的連接字符串，例如：

      • 名稱：DefaultConnection
      • 值：Server=myserver.database.windows.net;Database=logisticsdb;User Id=myuser;Password=mypassword;

    • 請確保名稱與應用程式代碼中的連接字符串名稱一致。

  2. 修改應用程式代碼來使用連接字符串：

3. 防火牆設置與訪問權限

當您將資料庫設置在雲端時，可能需要調整資料庫的防火牆規則，以允許您的 Azure App Service 實例連接到它。

  1. Azure SQL Database 的防火牆設置：

    • 登入 Azure 入口網站。
    • 找到您的 Azure SQL Database。
    • 點擊 「設定」，選擇 「防火牆和虛擬網路」。
    • 您可以添加一條規則，允許來自 Azure 服務的連接，或特定的 IP 地址段。
    • 您也可以選擇「允許 Azure 服務和資源訪問這個服務」來簡化連接設置。

  2. 允許 Azure App Service 訪問資料庫：

    • 在 Azure 入口網站中，找到 App Service 和資料庫的設置，並確保兩者的訪問權限設置正確，以便 App Service 能夠連接到資料庫。
    • 例如，在資料庫中允許 Azure 內部 IP 或直接允許應用程式服務連接。

4. 本地資料庫與雲端資料庫的連接

如果您的資料庫仍位於本地（例如開發期間），而您的 Web API 已部署到雲端，您需要確保本地資料庫對 Azure 開放連接：

  • VPN 連接：可以考慮使用 VPN 來將 Azure 服務與本地環境連接起來。
  • 動態 IP 或靜態 IP：確保您本地網絡的 IP 地址允許 Azure 的流量進入，並開啟資料庫埠號（如 MySQL 的 3306 埠或 SQL Server 的 1433 埠）。

5. 使用雲端資料庫的優點

將資料庫部署到雲端（例如 Azure SQL Database 或 Azure Database for MySQL），可以獲得以下優點：

  • 可擴展性：隨著應用的成長，您可以方便地擴展資料庫性能。
  • 高可用性：雲端資料庫提供內建的備份和高可用性選項，減少停機風險。
  • 維護方便：Azure 提供了資料庫的自動更新和修補服務，節省維護成本。

總結

部署 Web API 到雲端後，您可以通過配置雲端資料庫並設置正確的連接字符串來實現連接。建議將資料庫也部署到雲端，以便更好的性能、可擴展性以及方便的管理。如果有其他具體問題，隨時告訴我！

在 PostgreSQL 中有兩種方式來封裝和執行程式碼邏輯：

  1. 儲存函數（Stored Functions） - 在 PostgreSQL 早期版本中，您可以通過 CREATE FUNCTION 來創建儲存函數，它們能返回值，並且是主要用於執行計算或返回資料的。
  2. 儲存程序（Stored Procedures） - 從 PostgreSQL 11 版本開始，CREATE PROCEDURE 語法被引入，儲存程序允許通過 CALL 語句來執行，且支援交易控制（例如 COMMIT 和 ROLLBACK）。

儲存程序與儲存函數的區別

  • 儲存程序（Stored Procedures） 通過 CALL 語法執行，並且允許在執行期間執行 COMMIT 或 ROLLBACK 來控制事務，這使得它們在複雜交易中的應用更強大。
  • 儲存函數（Stored Functions） 必須返回一個值或數據集，並且不能控制交易，主要用於資料查詢或計算。

創建儲存程序的語法

以下是如何使用 PostgreSQL 中的 CREATE PROCEDURE 語法來創建一個儲存程序的例子：

  • **CREATE PROCEDURE**：創建儲存程序。
  • **IN tracking_number VARCHAR**：輸入參數，tracking_number 是包裹的追踪號。
  • **LANGUAGE plpgsql**：使用 PL/pgSQL 語言，它是 PostgreSQL 的一個過程語言，非常適合編寫複雜的邏輯。
  • **BEGIN ... END**：定義儲存程序的主體。

儲存函數的例子

如果您想創建一個儲存函數來計算某些值或返回查詢結果，您可以使用 CREATE FUNCTION。例如：

  • **CREATE FUNCTION**：創建儲存函數。
  • **RETURNS VARCHAR**：指定返回值類型。
  • **DECLARE**：用於聲明變量。
  • **RETURN**：返回計算結果。

交易控制的不同

  • 儲存函數 不允許在內部使用 COMMIT 或 ROLLBACK，因為函數中的語句總是作為一個整體在一個單獨的事務中執行。
  • 儲存程序 允許在程序內部進行 COMMIT 或 ROLLBACK，這使它非常適合用於需要進行部分提交的情況。

使用場景

  • 儲存函數 適合需要進行資料計算、查詢並返回結果的場景，像是業務邏輯計算、報表生成等。
  • 儲存程序 更適合包含多個步驟並需要交易控制的操作，像是批量更新、日誌寫入等。

總結

PostgreSQL 支援儲存程序和儲存函數，並且從 PostgreSQL 11 版本開始引入了新的 CREATE PROCEDURE 語法。儲存程序可以讓開發人員編寫更靈活的邏輯，特別是在需要控制事務的情況下。

  • 如果您需要返回某些計算結果或資料，使用 儲存函數。
  • 如果需要進行多步驟的操作並可能涉及交易控制，使用 儲存程序。

兩者的結合使用可以極大地提高資料庫邏輯的靈活性和性能。
