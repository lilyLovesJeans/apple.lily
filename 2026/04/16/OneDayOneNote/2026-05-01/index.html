2026-05-01 WEB API 也是OOP設計模式 , 它是什麼原因被隱藏? 被封裝?

非常好的問題！ASP.NET Core Web API 是基於面向物件編程（OOP）的設計模式，但在 ASP.NET Core 6 中的 最小化 API（Minimal API） 提供了一種新的、更簡潔的方式來構建 Web API。在這裡，Main 方法被「隱藏」和「封裝」的原因主要與框架的設計理念和開發者體驗的改進有關。讓我更詳細地說明。

1. 從 OOP 的角度來看 Web API

首先要了解，Web API 作為 ASP.NET Core 框架的一部分，是完全基於 OOP 設計的。這意味著它依然遵循著面向物件的基本原則，例如類別、屬性、方法、繼承和多態等。Web API 的架構本身是高度面向物件的，它由許多類別和接口組成來處理 HTTP 請求和回應。

不過，在應用的啟動過程中，框架的設計並不一定需要所有部分都完全暴露給開發者使用，因此在 ASP.NET Core 6 中，最小化 API 引入了隱藏部分啟動邏輯的方式。

2. 隱藏和封裝的原因

這裡的「隱藏」和「封裝」其實是一種框架設計上的選擇，目的是讓開發者能夠更高效地開發，並且專注於應用的主要邏輯，而不是樣板代碼（Boilerplate Code）和啟動å配置。具體來說：

2.1 簡化樣板代碼

在 ASP.NET Core 6 及以上版本中，最小化 API 的一個主要目的是減少樣板代碼。傳統的 ASP.NET Core Web API 需要有很多標準化的啟動代碼，例如定義 Main 方法、配置 Startup 類別、創建 Host 等等。

這些代碼雖然重要，但它們對於大部分項目來說是非常相似的，屬於樣板代碼。這些樣板代碼使得新手學習起來增加了困難，或者讓開發者在構建簡單的 API 時需要做許多重複工作。

因此，為了提高開發效率，框架將這些啟動邏輯進行了封裝和隱藏。這樣，開發者在編寫 API 時，不需要手動撰寫這些啟動代碼，而是直接撰寫應用程式的核心功能。

2.2 頂層語句的引入

C# 9 引入了頂層語句（Top-level Statements），這個新特性讓程式碼更加簡潔，特別是對於簡單應用和快速原型開發。使用頂層語句，可以直接撰寫應用程式的邏輯，而不需要顯式地撰寫 Main 方法。

  • 封裝啟動邏輯：當使用頂層語句時，編譯器自動把這些頂層語句包裝進一個 Main 方法中，從而封裝了啟動邏輯，讓開發者不必手動去寫這些樣板代碼。

2.3 提升開發者體驗

提升開發者體驗 是最小化 API 引入的主要目的之一。這種做法讓開發者可以在幾行代碼內快速啟動一個 Web API，從而專注於處理核心的應用邏輯。這種簡潔的風格對於新手和需要快速原型的場景非常有幫助。

2.4 對於複雜應用的彈性

雖然最小化 API 引入了這些改進，但並不意味著所有情況下都會隱藏 Main 方法。如果開發者需要對應用程式的啟動過程進行更多的控制，仍然可以自己定義 Main 方法，或者使用傳統的 Startup 類別來配置應用。這意味著 ASP.NET Core 保留了彈性，可以滿足簡單和複雜應用的需求。

3. 隱藏和封裝的背後：設計模式的應用

這裡的「隱藏」和「封裝」其實是一些設計模式的應用：

  • 封裝（Encapsulation）：框架對啟動邏輯進行封裝，讓開發者不需要關心啟動的細節，只需要專注於業務邏輯。
  • 模板方法模式（Template Method Pattern）：框架定義了應用啟動的模板，而開發者只需要提供具體的邏輯，框架會按照模板來運行應用。
  • 簡單工廠模式（Simple Factory Pattern）：WebApplication.CreateBuilder(args) 這樣的方法實際上是一種工廠方法，用來創建和配置應用程式。

這些設計模式的應用使得框架能夠提供一個清晰且一致的啟動過程，並且根據應用的需要自動處理許多細節。

小結

  • 在 ASP.NET Core 6 的 最小化 API 中，Main 方法被隱藏和封裝的原因主要是為了簡化樣板代碼，讓開發者可以更專注於應用的核心邏輯。
  • 頂層語句 和 框架封裝 使得 Web API 的啟動過程更為簡單，提高了開發效率 和 開發者體驗。
  • 這樣的隱藏是基於設計模式的應用，例如 封裝 和 模板方法，從而使得框架能夠自動處理啟動邏輯，減少開發者的工作負擔。
